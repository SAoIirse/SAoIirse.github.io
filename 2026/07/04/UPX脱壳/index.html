壳

什么是壳呢？壳大多分为两类：压缩壳和加密壳。

压缩壳

压缩壳早在 DOS 时代就已经出现，压缩壳可以帮助缩减 PE 文件的大小，隐藏PE文件内部代码和资源，便于传输和保存。压缩壳一种用于单纯的压缩普通PE文件，另一种则会对文件造成较大的改变，严重破坏PE文件头，用于压缩恶意程序。
常见的压缩壳有：Upx、ASpack、PECompat

加密壳

加密壳也称保护壳，应用有多种防止代码逆向分析的技术，主要的功能是保护 PE 免受代码逆向分析。加密壳保护的 PE 程序通常比原文件大得多。目前加密壳大量用于对安全性要求高，对破解敏感的应用程序，同时也有恶意程序用于避免（降低）杀毒软件的检测查杀。

常见的加密壳有：ASProtector、Armadillo、EXECryptor、Themida、VMProtect

壳加载的过程：

保存入口参数：保持栈平衡，壳运行前后要保持寄存器值不变

获取程序API：

  1. 一般壳的输入表中只有 GetProcAddress、GetModuleHandle 和 LoadLibrary 这几个 API 函数
  2. 如果需要其他 API 函数，则通过 LoadLibraryA(W) 或 LoadLibraryExA(W) 将 DLL 文件映射到调用进程的地址空间中
  3. 如果 DLL 文件已被映射到调用进程的地址空间里，可以调用 GetModuleHandleA获得 DLL 模块句柄
  4. 一旦 DLL 模块被加载，就可以调用 GetProcAddress 函数获取输入函数的地址

解密各区块数据

  1. 为达到保护源程序代码和数据的目的，一般会加密源程序文件的各个区块。在程序执行时外壳将这些区块数据解密，来让程序正常运行
  2. 外壳一般按区块加密，按区块解密，并将解密的数据放回在合适的内存位置

跳转回原程序入口点

  1. 在跳转回入口点之前，一般会恢复填写原 PE 文件输入表，并处理重定位项
  2. 因为加壳时外壳自己构造了一个输入表，因此在这里需要重新对每一个 DLL 引入的所有函数重新获取地址，并填写到 IAT 表中
  3. 做完上述工作后，会将控制权移交原程序，并继续执行

UPX壳

加载原理：压缩区段 + 运行时解压

UPX 的设计核心在于对 PE 文件结构的重组和一段自解压代码。

1.结构重组与压缩
UPX 首先对原始 PE 文件的各区段（如代码段 .text、数据段 .data）进行高比例压缩，并将压缩后的内容合并存入一个新区段 UPX1。同时，它会在文件中预留另一个区段 UPX0，该区段在磁盘上大小通常为 0，但被声明为在内存中拥有完整展开空间，用于后续存放解压后的原始代码与数据。对于资源段 .rsrc，UPX 一般保留原样或仅做轻微处理，以便系统在不解压的情况下仍能正常读取图标等资源信息。

2.自解压代码（Stub）接管入口
加壳后，程序的入口点（OEP）被修改，指向 UPX 附加在文件末尾的自解压代码（Stub）。当程序被加载执行时，系统首先运行的并非原始程序逻辑，而是这段 Stub 代码，从而开始解压流程。

3.现场保护与解压修复
Stub 代码开始执行时，首先执行 pushad 指令，将所有通用寄存器的当前值压入堆栈，以保存加壳前的“现场”状态。随后，Stub 将 UPX1 区段中的压缩数据解压到预先预留的 UPX0 内存区域，并在此过程中完成必要的 PE 结构修复工作，包括重建导入地址表（IAT）等，确保解压后的程序能正确调用外部 API。

4.恢复现场与权利工作移交
解压及初始化工作全部完成后，Stub 执行 popad 指令，从堆栈恢复所有通用寄存器的原始值，使 CPU 状态回到加壳前的模样。最后，通过一条远跳转指令（jmp）跳转到原始程序的真正入口点（OEP），将执行权完整、干净地移交给原始程序，至此加壳程序完成使命，原始逻辑开始运行。

未运行时区段状态：

● UPX0 区段：原始数据大小（VirtualSize）很大，比如 0x19000，但在磁盘上的大小（RawSize）几乎为 0 或很小。
● UPX1 区段：占据了文件的大部分体积，包含了经过 LZMA 或 NRV 算法高强度压缩后的原始程序代码和数据。
● .rsrc 区段：通常保持原样，存放图标、版本信息等资源，以便操作系统在不完全解压时也能读取。

运行时内存动态变化

Step 1: 系统加载与 Stub 接管 (对应流程图 A -> B)
● 变化：操作系统将上述三个区段装载到内存中。
○ UPX0 被分配了一块指定大小的空内存页（全是 00 或随机数据）。
○ UPX1 被完整载入，包含压缩数据和 Stub 代码。
● CPU 视角：EIP（指令指针）直接跳转到 UPX1 区段内的 Stub 代码开始执行。
Step 2: 保护现场 (对应流程图 B1)
● 变化：执行 pushad。
● 内存表现：所有通用寄存器的值（EAX, ECX, EDX, EBX, ESP, EBP, ESI, EDI）被依次压入当前堆栈。
● 目的：这是在执行解压这个”脏活累活”前，把原始程序即将用到的寄存器环境备份起来。
Step 3: 解压循环 (对应流程图 B2)
● 变化：这是内存变动最剧烈的阶段。
○ 读取：CPU 从 UPX1 区段读取压缩数据流。
○ 解算：Stub 中的解压算法实时解算数据。
○ 写入：将解压后的原始机器码写入到 UPX0 区段预留的空内存页中。
● 核心逻辑：磁盘上空无一物的 UPX0 此刻被动态填充，变回了原本的 .text (代码段) 和 .data (数据段)。
Step 4: 修复导入表 (对应流程图 B3)
● 变化：写入 UPX0 的数据中，调用外部 DLL 函数的地址（IAT）是不对的。Stub 会解析原程序的导入表结构，调用 GetProcAddress 获取真实 API 地址，并覆盖写入到 UPX0 对应的内存位置。
Step 5: 恢复现场与交接 (对应流程图 B4 -> C)
● 变化：
a. 执行 popad。刚才压在栈底的原始寄存器值被弹回 CPU，CPU 的环境恢复如初。
b. 执行 jmp OEP。CPU 的指令指针 EIP 离开 UPX1，跨区段跳入 UPX0。
● 最终状态：此时内存中，UPX0 已经变成了完全可运行的原始程序，CPU 开始从原始入口点执行代码。UPX1 里的 Stub 代码使命完成，处于”闲置”状态。

总结

理解上述内存变化后我们就能理解动态脱壳的时候，我们要找什么了！
1.找到写入完成的瞬间：upx0数据写入完成，寄存器恢复（popad之后）
2.找到跨区段跳跃的瞬间：EIP从UPX1跳转到UPX0时的jmp指令。这个跳转目标就是我们要dump的原始程序的入口点

单步跟踪法 – 万能但是工作量巨大

单步跟踪法的核心，是利用调试器（如 Ollydbg）的单步步入（F7）、单步步过（F8）和运行到选定位置（F4）功能，逐条或分段执行加壳程序的自解压代码，在避开复杂循环的同时确保不遗漏关键跳转，最终在解压完成后准确抵达原始程序入口点（OEP），并进行内存转储（Dump）。

具体操作：

1.以步过为主，优先实现跨段跳转
程序加载后，首先使用 F8（步过） 单步向下执行。遇到向下的 jmp 跳转指令时，优先让其跳转，以快速推进执行流，避免停留在当前代码块中。

2.遇到大循环时，用“运行到选定位置”跳过
单步过程中经常会遇到重复次数多、逻辑复杂的大型循环。此时不要在循环中逐条单步，而是将光标移动到循环体之后的指令行，按 F4（运行到选定位置），让程序直接执行完循环并停在你指定的下一行，从而高效跳过循环干扰。

3.近 Call 指令主动进入，防止错过关键解码
当遇到近调用（call 指令，且目标地址较近）时，尽量不要用 F8 直接步过，而是改用 F7（步入） 进入该 call 内部执行。因为很多加壳程序会将部分解压或修复代码放在此类 call 中，直接跳过可能遗漏重要初始化步骤，导致后续无法正确抵达 OEP。

4.关注大幅度跳转，定位 OEP
在执行过程中，凡是遇到跳转跨度很大（例如从当前区段跳向另一个区段，或跳向 UPX0 等区段）的 jmp 指令，此跳转极有可能就是自解压代码结束、跳往原始 OEP 的最终转移。遇到此类跳转时需格外留意，单步跟踪进去，即可到达原始程序的真正入口点。

ESP定律 – 堆栈平衡定律

原理

利用 pushad / popad 的对称性定位 OEP
pushad 指令会将所有通用寄存器的值压入堆栈，而对应的 popad 指令一定会从同一堆栈位置恢复这些值。因此，pushad 执行时写入的栈地址，必然会被 popad 再次访问。在该栈地址上设置硬件断点（硬件访问断点），即可在 popad 执行的那一刻精准中断——这个时刻恰好是解压完成、即将跳往原始入口点（OEP）之前的最近位置。

具体操作：

1.载入加壳程序后，从入口开始单步执行，步过（Step Over）每一条指令，直到执行到 pushad 指令。
执行完 pushad 后，观察堆栈窗口，找到栈顶指针 ESP 所指向的地址（此时该地址处存放的正是刚刚压入的第一个寄存器的值）。
2.右键点击该栈地址，选择 硬件断点（Hardware Breakpoint） → 访问断点（Access）（或写断点，视调试器选项而定）。
3.设置好断点后，按 F9 运行程序。调试器将在 popad 访问该栈地址时自动中断，此时程序已经完成解压与修复，正处于 popad 指令执行后、远跳转到 OEP 之前的关键位置，随后单步跟入跳转即可到达原始 OEP。

这里解释为什么要用HW硬件断点，而不是普通断点？
首先我们要明白，普通断点可以修改代码字节但容易被壳检测到，所以普遍适用于静态代码位置。而硬件断点是利用CPU调试寄存器，不修改代码，比较隐蔽不容易被壳检测到，所以大多需要动态监控内存和对栈地址访问时，使用HW断点。

注意事项

● 不是所有壳都适用：ESP定律对UPX、ASPack等压缩壳非常有效，但对一些强壳（如VMProtect、Themida）可能无效，因为它们不会用简单的 pushad / popad 模式。
● ESP变红不是唯一信号：有些壳用 pushfd / popfd 而不是 pushad，原理相同，注意观察即可。
● 设断点时机：在ESP刚刚变红的那一刻设断点，而不是等执行了好多步之后。

内存镜像法

内存镜像法是我们在加壳程序被加载时,进入到程序虚拟内存区段. 通过加两次内存一次性断点, 到达程序正确 OEP 的位置.内存镜像法的原理在于对程序资源段和代码段下断点,具体要多了解一下PE壳。

此处解答为什么要对TEXT和RSRC两段下断点？

因为，程序运行时会解压原始代码到.text，而资源.rsrc是在最后才被解压。当程序访问到资源段的时候就说明程序已经被解压完成，之后需要跳转回到text段。

重要步骤：

  1. 选择菜单的选项->调试选项->异常
  2. 勾选所有的忽略异常
  3. 按下ALT+M, 打开内存镜像, 找到程序的第一个.rsrc, 按 F2 下断点, 然后按SHIFT+F9运行到断点
  4. 再按ALT+M, 打开内存镜像, 找到程序的第一个.rsrc上面的.text, 按 F2 下断点. 然后按SHIFT+F9

脱壳之后

我们会发现脱壳之后程序是运行不起来的，会显示如下图报错

说明脱壳后的程序无法正确找到或执行其导入的函数。核心原因是 OllyDump 的“重建输入表”功能对 UPX 壳没有完美工作

如何正确修复？

使用 ImportREC

对于 UPX 这种压缩壳，我们放弃 OllyDump 的自动修复功能，采用标准的“两步走”：先 Dump，后手动修复。原理本质：把函数对应表中的名字修复回地址，使其能够运行
第一步：用 OllyDump 再次 Dump，但这次不勾选“重建输入表”

  1. 确认你仍处于 OEP 处（004010CC）。
  2. 打开 OllyDump，这次取消勾选“重建输入表”。
  3. 点击“脱壳”，将文件另存为，例如 dump.exe。这个文件目前是“残废”的，需要下一步来修复。
    第二步：使用 ImportREC 手动修复导入表
  4. 打开 ImportREC：建议以管理员身份运行。
  5. 附加进程：在 ImportREC 顶部的下拉列表中，选中你正在调试的 NotePad.exe 进程。
  6. 填写 OEP 信息：
    ○ 在 OEP 输入框中，填入 000010CC。
    ○ 点击 IAT Autosearch 按钮。ImportREC 通常能自动找到导入表。
    ○ 如果成功，会弹窗提示“发现信息…”，点击确定。再点击 Get Imports 按钮，它会列出所有导入的函数。
  7. 修复文件：
    ○ 点击 Fix Dump 按钮。
    ○ 在弹出的对话框中，选择你刚才 Dump 出来的 dump.exe 文件。
  8. 得到脱壳文件：ImportREC 会生成一个名为 dump_.exe 的文件，这个就是修复好导入表、可以正常运行的脱壳程序。
